Python

Python is dynamically typed and interpreted: names are just labels bound to objects, and an object carries its own type. Most of what you need for interviews is fluency with the built-in container types, comprehensions, and a handful of standard-library tools. The mental model to hold onto: everything is an object, variables are references, and "mutable vs immutable" decides whether an operation copies or modifies in place.

Core types & the object model

Every value is an object with an identity (id()), a type (type()), and a value. The single most exam-relevant distinction is mutability: immutable objects (int, float, str, tuple, frozenset, bytes) can never change — "modifying" one actually creates a new object — while mutable objects (list, dict, set, bytearray) can be changed in place. This is why passing a list into a function can change the caller's list, but passing an int cannot.

Identity vs equality. == compares values; is compares identity (same object in memory). Use is only for singletons like None: write if x is None, never if x == None.

Truthiness. Empty containers ([] {} () "" set()), 0, 0.0, and None are falsy; almost everything else is truthy. So if items: is the idiomatic "is this list non-empty?" check.

Numbers & operators

Integers are arbitrary precision (no overflow), so factorials and big sums just work. Division always has two flavours you must keep straight: / is true division returning a float, // is floor division. Floats are IEEE-754 doubles, so equality comparisons are unreliable — compare with a tolerance instead.

python
7 / 2      # 3.5   true division -> float
7 // 2     # 3     floor division
-7 // 2    # -4    floors toward negative infinity (not 0!)
7 % 3      # 1     modulo; sign follows the divisor in Python
2 ** 10    # 1024  exponent
divmod(7, 3)        # (2, 1)  quotient and remainder together

import math
math.isclose(0.1 + 0.2, 0.3)   # True  -- never use == on floats
round(2.675, 2)                # 2.67 (binary float surprise)
abs(-5); math.floor(2.9); math.ceil(2.1); math.sqrt(16)
float('inf'); float('nan')     # special values
Bit tricks for interviews. x & 1 tests odd/even, x << n multiplies by \(2^n\), x >> n divides, x & (x-1) clears the lowest set bit (count bits / check power of two), x ^ y finds the single non-duplicated value in a list.

Strings

Strings are immutable sequences, so every "edit" returns a new string. That immutability is why building a string in a loop with += is quadratic — accumulate into a list and "".join() at the end instead. f-strings are the modern way to format and also support a debug = form and format specs.

python
s = "Data Science"
s.lower(); s.upper(); s.title(); s.strip()
s.replace("Data", "Computer")
s.split()              # ['Data', 'Science']   split on whitespace
"a,b,c".split(",")     # ['a','b','c']
",".join(["a","b"])    # 'a,b'   <-- preferred way to build strings
s.startswith("Data"); s.endswith("ce"); "Sci" in s
s.find("Sci")          # 5  (-1 if absent); .index raises if absent
s[0]; s[-1]; s[:4]; s[::-1]   # indexing & slicing (reverse!)

name, score = "Ada", 0.97
f"{name} scored {score:.1%}"   # 'Ada scored 97.0%'
f"{score=}"                     # 'score=0.97'  debug form
"  pad ".rjust(8); "x".zfill(3)  # '00x'
ord('A'); chr(65)                # 65 ; 'A'
Common bug. str.replace/strip/sort… do not mutate — they return a new value. s.strip() alone does nothing; you must write s = s.strip().

Lists

A list is a dynamic array: indexing and appending are O(1) amortized, but inserting or deleting at the front is O(n) because everything shifts. Know which methods mutate in place (and return None) versus which return a new object — mixing this up is the most common silent bug in interview code.

Tuples & sequence protocol

Tuples are immutable lists, which makes them hashable — that's why they (not lists) can be dictionary keys or set members. They're also the natural type for returning multiple values and for fixed records. Named tuples and dataclasses give you readable field access without writing boilerplate.

python
point = (3, 4)
x, y = point                    # unpacking
single = (5,)                   # one-element tuple needs the comma

from collections import namedtuple
Pt = namedtuple("Pt", "x y")
p = Pt(3, 4); p.x; p.y          # readable, still immutable

from dataclasses import dataclass
@dataclass
class User:
    name: str
    age: int = 0                # auto __init__, __repr__, __eq__

Dictionaries

The dict is Python's hash table and your go-to for O(1) lookups, counting, grouping, caching, and adjacency lists. Since 3.7 it preserves insertion order. The key skills: safe access that won't raise, and the counting/grouping idioms that appear in almost every data interview.

python
d = {"a": 1, "b": 2}
d["c"] = 3                      # insert / update
d.get("z", 0)                   # 0 if missing -- no KeyError
d.setdefault("a", 99)           # returns 1; only sets if absent
d.pop("a", None); del d["b"]
"a" in d                        # key membership, O(1)
d.keys(); d.values(); d.items() # views
for k, v in d.items(): ...
{**d1, **d2}                    # merge (d2 wins);  d1 | d2 in 3.9+

# Counting & grouping -- memorize these
from collections import Counter, defaultdict
Counter("mississippi").most_common(2)   # [('i',4),('s',4)]
groups = defaultdict(list)
for word in words:
    groups[len(word)].append(word)       # auto-creates the list

# Dict comprehension & inversion
squares = {n: n*n for n in range(5)}
inv = {v: k for k, v in d.items()}
Counter is a multiset. It supports + - & | between counters and .most_common(), making "top-k frequent" a two-liner.

Sets

Sets give O(1) membership and automatic deduplication, plus the algebra of union/intersection/difference. Whenever a problem says "have I seen this before?" or "what's common between these?", a set is usually the answer and often turns an O(n²) scan into O(n).

python
s = {1, 2, 3}; empty = set()    # {} is an empty DICT, not a set
s.add(4); s.discard(9)          # discard won't raise; remove will
2 in s                          # O(1) membership
a, b = {1,2,3}, {2,3,4}
a | b      # union {1,2,3,4}
a & b      # intersection {2,3}
a - b      # difference {1}
a ^ b      # symmetric difference {1,4}
a <= b     # subset test
unique = set(my_list)           # dedup in one call
frozenset({1,2})                # immutable -> hashable

Comprehensions & functional tools

Comprehensions are the Pythonic replacement for building a collection with a loop: they're faster and read as "what I want, for each item, where condition". The same syntax produces lists, sets, dicts, and (with parentheses) lazy generators. Pair them with map/filter/zip/enumerate and you can express most transforms in one line.

python
[x*x for x in range(10) if x % 2 == 0]        # list
{x % 3 for x in range(10)}                    # set
{x: x*x for x in range(5)}                    # dict
(x*x for x in range(10**9))                   # generator: lazy, O(1) memory
[[r*c for c in range(3)] for r in range(3)]   # nested / 2D
[y for row in matrix for y in row]            # flatten (order = nested loops)
[a if a > 0 else 0 for a in vals]             # ternary inside

# Iteration helpers
for i, v in enumerate(seq, start=1): ...      # index + value
for a, b in zip(xs, ys): ...                  # parallel iterate
list(zip(*matrix))                            # transpose!
from itertools import product, combinations, permutations, accumulate, chain, groupby
list(combinations([1,2,3], 2))                # [(1,2),(1,3),(2,3)]
list(accumulate([1,2,3,4]))                   # running sum [1,3,6,10]

Functions, scope & arguments

Functions are first-class objects: you can pass them around, return them, and store them. Master the argument forms — positional, keyword, defaults, *args, **kwargs — and the LEGB scope rule (Local → Enclosing → Global → Built-in) that decides which name a lookup finds. Closures capture enclosing variables by reference, which is the basis of decorators.

python
def f(a, b=10, *args, key=None, **kwargs):
    # a: required, b: default, args: extra positionals (tuple),
    # key: keyword-only, kwargs: extra keywords (dict)
    return a + b

f(1, 2, 3, key="x", z=9)
nums = [1, 2, 3]
print(*nums)                 # unpack into positionals -> 1 2 3
f(**{"a": 1, "b": 2})        # unpack dict into keywords

# Closures & decorators
def timer(fn):
    import functools, time
    @functools.wraps(fn)
    def wrapper(*a, **k):
        t = time.perf_counter(); r = fn(*a, **k)
        print(f"{fn.__name__}: {time.perf_counter()-t:.4f}s")
        return r
    return wrapper

@timer
def slow(): ...              # slow = timer(slow)

square = lambda x: x*x       # anonymous function
sorted(data, key=lambda r: (r.age, r.name))   # multi-key sort
LEGB & global/nonlocal. Assigning to a name inside a function makes it local unless you declare global x (module scope) or nonlocal x (enclosing function). Reading without assigning falls through the LEGB chain.

Classes & OOP

A class bundles state (attributes) with behaviour (methods). Instance methods take self; __init__ is the constructor; dunder methods let your objects behave like built-ins (__len__, __eq__, __repr__, __lt__ for sorting). The four OOP pillars — encapsulation, inheritance, polymorphism, abstraction — show up as conceptual interview questions, so connect each to a concrete Python feature.

python
class Animal:
    species = "unknown"               # class attribute (shared)
    def __init__(self, name):
        self.name = name              # instance attribute
    def speak(self):                  # to be overridden
        raise NotImplementedError
    def __repr__(self):
        return f"Animal({self.name!r})"

class Dog(Animal):                    # inheritance
    species = "dog"
    def speak(self):                  # polymorphism / override
        return "Woof"
    def fetch(self):
        return f"{self.name} fetches"

d = Dog("Rex")
d.speak(); isinstance(d, Animal)      # True
super()      # call parent method inside an override

class Circle:
    def __init__(self, r): self._r = r
    @property                         # computed read-only attribute
    def area(self): return 3.14159 * self._r**2
    @staticmethod
    def unit(): return Circle(1)      # no self/cls
    @classmethod
    def from_diameter(cls, d): return cls(d/2)   # alt constructor

Iterators & generators

Iteration is a protocol: an iterable can produce an iterator via iter(), and an iterator yields items via next() until it raises StopIteration. Generators (functions with yield) build iterators lazily — they produce values on demand, so you can stream gigabytes or model infinite sequences in constant memory. This laziness is the key to scalable ETL in pure Python.

python
def countdown(n):
    while n > 0:
        yield n            # pauses here, resumes on next()
        n -= 1

for x in countdown(3): print(x)     # 3 2 1
gen = (line.strip() for line in open("big.csv"))   # streams, O(1) memory

import itertools as it
it.islice(gen, 10)         # take first 10 without loading all
it.chain(a, b)             # concatenate iterables lazily
next(iter(d))              # first key of a dict
Why it matters for data engineering. Reading a file with a generator processes one row at a time instead of loading the whole file — the difference between a pipeline that scales and one that OOMs.

Errors, exceptions & context managers

Python prefers "easier to ask forgiveness than permission" (EAFP): try the operation and catch the specific exception, rather than checking every precondition first. Catch the narrowest exception you can, use finally for cleanup, and prefer with blocks (context managers) so resources like files and locks are released automatically even on error.

python
try:
    val = int(user_input)
except ValueError as e:
    print("not a number:", e)
except (KeyError, IndexError):
    ...
else:
    print("ok:", val)        # runs only if no exception
finally:
    cleanup()                # always runs

raise ValueError("bad input")          # raise your own
assert x > 0, "x must be positive"     # debugging checks

with open("data.csv") as f:            # auto-closes the file
    data = f.read()

from contextlib import contextmanager
@contextmanager
def timer():
    import time; t=time.perf_counter()
    yield
    print(time.perf_counter()-t)

Interview gotchas & idioms

Mutable defaults persist across calls. Use None sentinels.
[[0]*n]*m makes shared rows. Use a comprehension.
is vs ==: is only for None/singletons.
Float equality: use math.isclose, never ==.

Big-O & complexity analysis

Big-O describes how runtime or memory grows as input \(n\) grows, ignoring constants and lower-order terms — it answers "will this still work when the data is 100× bigger?". You analyse the worst case unless told otherwise. The hierarchy below is worth knowing cold, because naming the complexity of your solution (and the brute force you improved on) is half of what's being scored.

NotationNameExamplen=10⁶ feels like
\(O(1)\)constantdict/array index, hash lookupinstant
\(O(\log n)\)logarithmicbinary search, balanced-tree op~20 steps
\(O(n)\)linearsingle scan, one passfine

Amortized vs worst case. A list.append is O(1) amortized: most appends are cheap, and the rare resize that copies everything averages out. Always also state space complexity (extra memory), including the recursion stack — a recursion of depth \(n\) uses \(O(n)\) space even if it allocates nothing.

Arrays & the two-pointer / sliding-window family

When data is a sequence and you'd otherwise compare every pair (O(n²)), two pointers or a sliding window usually collapse it to O(n). Two pointers walk inward from both ends (works on sorted data: pair-sum, palindrome) or at different speeds. A sliding window expands a right edge and shrinks a left edge to maintain a constraint over contiguous subarrays. A prefix sum precomputes cumulative totals so any range sum is an O(1) subtraction.

python
# Two pointers: pair summing to target in a SORTED array
def two_sum_sorted(a, target):
    i, j = 0, len(a) - 1
    while i < j:
        s = a[i] + a[j]
        if s == target: return (i, j)
        if s < target:   i += 1     # need bigger -> move left up
        else:            j -= 1     # need smaller -> move right down
    return None

# Sliding window: longest substring without repeats
def longest_unique(s):
    seen, left, best = {}, 0, 0
    for right, ch in enumerate(s):
        if ch in seen and seen[ch] >= left:
            left = seen[ch] + 1     # shrink window past the repeat
        seen[ch] = right
        best = max(best, right - left + 1)
    return best

# Prefix sums: range sum in O(1) after O(n) build
pre = [0]
for x in a: pre.append(pre[-1] + x)
range_sum = pre[r+1] - pre[l]       # sum of a[l..r]

Hash maps & sets — the O(1) trade

A hash table trades memory for speed: it spends O(n) extra space to turn "have I seen this / what's paired with this" from an O(n) scan into an O(1) lookup. The classic move is one pass that remembers what you've seen so the answer is ready when you reach its partner — this is how Two Sum drops from O(n²) to O(n).

python
# Two Sum, unsorted, O(n): remember complements as you go
def two_sum(nums, target):
    seen = {}                       # value -> index
    for i, x in enumerate(nums):
        if target - x in seen:
            return (seen[target - x], i)
        seen[x] = i

# Group anagrams: sorted letters as a hashable key
from collections import defaultdict
groups = defaultdict(list)
for w in words:
    groups[tuple(sorted(w))].append(w)
Heuristic: if you catch yourself writing a nested loop to find a relationship between elements, ask whether a hash map can remember the inner loop's work instead.

Stacks & queues

A stack is LIFO (last in, first out) — the right model for nesting, matching, undo, and DFS. A queue is FIFO (first in, first out) — the model for level-order/BFS and scheduling. In Python a list is a fine stack (append/pop), but for a queue use collections.deque so popping the front is O(1) instead of O(n).

python
# Stack: valid parentheses
def valid(s):
    pairs = {')':'(', ']':'[', '}':'{'}
    stack = []
    for ch in s:
        if ch in "([{": stack.append(ch)
        elif not stack or stack.pop() != pairs[ch]:
            return False
    return not stack

# Monotonic stack: next greater element (O(n))
def next_greater(a):
    res, stack = [-1]*len(a), []
    for i, x in enumerate(a):
        while stack and a[stack[-1]] < x:
            res[stack.pop()] = x
        stack.append(i)
    return res

from collections import deque
q = deque(); q.append(x); q.popleft()   # O(1) FIFO queue

Linked lists

A linked list stores each value in a node that points to the next, so inserting or deleting given a node is O(1) — but there's no random access, so reaching the k-th element is O(n). Interview value is mostly in pointer manipulation: reversing, detecting cycles with fast/slow pointers (Floyd's), and finding the middle in one pass.

python
class Node:
    def __init__(self, val, nxt=None): self.val, self.next = val, nxt

def reverse(head):                  # iterative reversal, O(n)/O(1)
    prev = None
    while head:
        head.next, prev, head = prev, head, head.next
    return prev

def has_cycle(head):                # Floyd's tortoise & hare
    slow = fast = head
    while fast and fast.next:
        slow, fast = slow.next, fast.next.next
        if slow is fast: return True
    return False

Heaps / priority queues

A binary heap keeps the smallest (or largest) element instantly available while supporting O(log n) insert and remove — perfect when you repeatedly need "the next best" item: top-k, merging sorted streams, Dijkstra, scheduling, running medians. Python's heapq is a min-heap on a plain list; negate values (or use tuples) for a max-heap.

python
import heapq
# Top-k largest with a size-k MIN-heap -> O(n log k), O(k) space
def top_k(nums, k):
    heap = []
    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k: heapq.heappop(heap)   # drop the smallest
    return heap
heapq.nlargest(k, nums)             # built-in shortcut

# Max-heap via negation
maxheap = []; heapq.heappush(maxheap, -x); top = -heapq.heappop(maxheap)
Top-k rule of thumb. For the k largest, keep a min-heap of size k (its root is the weakest survivor to evict). For the k smallest, keep a max-heap of size k. This is O(n log k), beating a full O(n log n) sort when k ≪ n.

Trees & binary search trees

Trees model hierarchy. A binary search tree keeps left < node < right, giving O(log n) search/insert/delete when balanced (and O(n) when degenerate into a list). The three depth-first traversals differ only in when you visit the node relative to its children — and in-order traversal of a BST yields sorted order, a fact interviewers test constantly.

class TreeNode:
    def __init__(self, val, left=None, right=None):
        self.val, self.left, self.right = val, left, right

def inorder(node, out):             # sorted for a BST
    if node:
        inorder(node.left, out)
        out.append(node.val)
        inorder(node.right, out)

def max_depth(node):                # height of a tree
    if not node: return 0
    return 1 + max(max_depth(node.left), max_depth(node.right))

from collections import deque
def level_order(root):              # BFS by levels
    if not root: return []
    q, res = deque([root]), []
    while q:
        level = []
        for _ in range(len(q)):     # fix the level boundary
            n = q.popleft(); level.append(n.val)
            if n.left:  q.append(n.left)
            if n.right: q.append(n.right)
        res.append(level)
    return res

Graphs: BFS & DFS

A graph is nodes plus edges; represent it as an adjacency list (dict of neighbours) for sparse graphs. BFS explores level by level with a queue and finds the shortest path in an unweighted graph; DFS dives deep with a stack/recursion and suits cycle detection, topological sort, and connected components. The one non-negotiable: track visited or you'll loop forever.

python
from collections import deque
graph = {0:[1,2], 1:[2], 2:[0,3], 3:[]}   # adjacency list

def bfs(start):                     # shortest hops, unweighted
    seen, q = {start}, deque([start])
    while q:
        node = q.popleft()
        for nb in graph[node]:
            if nb not in seen:
                seen.add(nb); q.append(nb)
    return seen

def dfs(node, seen=None):
    seen = seen or set()
    seen.add(node)
    for nb in graph[node]:
        if nb not in seen: dfs(nb, seen)
    return seen

# Grid problems: treat each cell as a node, 4-directional neighbours
DIRS = [(1,0),(-1,0),(0,1),(0,-1)]
Weighted shortest paths. BFS only works when every edge costs the same. With positive weights use Dijkstra (a heap-driven BFS); with possible negative edges use Bellman-Ford; for all-pairs use Floyd-Warshall.

def binary_search(a, target):       # a is sorted; O(log n)
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if a[mid] == target: return mid
        if a[mid] < target:  lo = mid + 1
        else:                hi = mid - 1
    return -1

Recursion & backtracking

Recursion solves a problem by reducing it to smaller instances of itself; every recursion needs a base case (when to stop) and a recursive case that moves toward it. Backtracking is recursion that builds a candidate incrementally and abandons ("prunes") a branch the moment it can't lead to a valid solution — the standard tool for permutations, combinations, subsets, N-queens, and Sudoku.

python
# Backtracking template: all subsets
def subsets(nums):
    res = []
    def backtrack(start, path):
        res.append(path[:])             # record current choice
        for i in range(start, len(nums)):
            path.append(nums[i])        # choose
            backtrack(i + 1, path)      # explore
            path.pop()                  # un-choose (backtrack)
    backtrack(0, [])
    return res
Stack limit. Python caps recursion (~1000 deep). For deep/linear recursion convert to iteration with an explicit stack, or raise sys.setrecursionlimit cautiously.

Dynamic programming

DP applies when a problem has overlapping subproblems (the same smaller question recurs) and optimal substructure (the best overall answer is built from best sub-answers). You either memoize top-down (recursion + cache) or build a table bottom-up. The hard part is defining the state and recurrence; once you can write "let dp[i] = the answer for the first i items", the transition usually follows.

python
# Top-down memoization (fib): exponential -> O(n)
from functools import lru_cache
@lru_cache(None)
def fib(n): return n if n < 2 else fib(n-1) + fib(n-2)

# Bottom-up table: climbing stairs (= fib in disguise)
def climb(n):
    dp = [0]*(n+1); dp[0] = dp[1] = 1
    for i in range(2, n+1):
        dp[i] = dp[i-1] + dp[i-2]       # reach i from i-1 or i-2
    return dp[n]

# 0/1 knapsack: dp[w] = best value within capacity w
def knapsack(weights, values, cap):
    dp = [0]*(cap+1)
    for wt, val in zip(weights, values):
        for w in range(cap, wt-1, -1):  # iterate DOWN for 0/1
            dp[w] = max(dp[w], dp[w-wt] + val)
    return dp[cap]
Spotting DP. Phrases like "count the number of ways", "minimum/maximum cost to", or "can you reach" over choices usually signal DP. First write the brute-force recursion, then add a cache — that is top-down DP.

The logical order of execution

You write SELECT first, but the engine evaluates it nearly last. This single table explains most SQL "why doesn't this work" questions: column aliases defined in SELECT aren't visible to WHERE (it runs earlier), and you filter aggregates with HAVING (after grouping) not WHERE (before).

#ClauseWhat it does
1FROM / JOINChoose & combine source tables
2WHEREFilter individual rows (no aggregates yet)
3GROUP BYCollapse rows into groups
4HAVINGFilter groups (can use aggregates)
5SELECTPick columns, compute expressions & aliases
6DISTINCTRemove duplicate rows
7ORDER BYSort (can use SELECT aliases)
8LIMIT / OFFSETTake a slice of the result
Mnemonic consequence. Want to filter on a computed alias? You can in ORDER BY (step 7) but not in WHERE (step 2) — wrap it in a subquery/CTE or repeat the expression.

SELECT & filtering

sql
SELECT customer_id,
       amount * 1.2 AS amount_with_tax        -- expression + alias
FROM   orders
WHERE  status = 'paid'
  AND  created_at >= '2026-01-01'

NULL is not a value, it's "unknown". x = NULL is never true — use IS NULL / IS NOT NULL. Any arithmetic or comparison with NULL yields NULL, and NOT IN (… NULL …) can silently return no rows.

Joins

A join combines rows from two tables on a matching condition. The join type decides what happens to rows that have no match: INNER keeps only matches, LEFT keeps all left rows (NULLs where the right has no match), RIGHT the mirror, FULL keeps everything. Picking the wrong one — usually INNER when you needed LEFT — silently drops rows and is the most common data-pipeline bug.

SELECT c.name, COUNT(o.id) AS n_orders
FROM   customers c
LEFT JOIN orders o ON o.customer_id = c.id    -- keep customers w/ 0 orders
GROUP BY c.name;

-- Find customers with NO orders (anti-join)
SELECT c.id
FROM   customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE  o.id IS NULL;                          -- the "missing" pattern

Fan-out trap. Joining to a table with multiple matching rows multiplies your rows, inflating any SUM/COUNT. Aggregate the child table first (in a CTE) before joining, or count DISTINCT.

Aggregation & GROUP BY

Aggregate functions collapse many rows into one summary. GROUP BY defines the buckets; every non-aggregated column in SELECT must appear in GROUP BY. Remember COUNT(*) counts rows (including NULLs) while COUNT(col) skips NULLs, and that HAVING filters the grouped result.

       COUNT(*)            AS n_orders,
       COUNT(DISTINCT customer_id) AS n_customers,

       AVG(amount)         AS avg_order,
       MAX(amount)         AS biggest
FROM   orders
WHERE  status = 'paid'                 -- pre-aggregation filter
GROUP BY region
HAVING SUM(amount) > 10000              -- post-aggregation filter
ORDER BY revenue DESC;

Window functions

Window functions compute across a set of rows related to the current row without collapsing them — you keep every row and get an aggregate, ranking, or running total beside it. The anatomy is func() OVER (PARTITION BY … ORDER BY …): PARTITION BY resets the calculation per group, ORDER BY defines sequence for rankings and running sums. These are the single highest-leverage SQL topic for analytics interviews.

  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY customer_id
                            ORDER BY created_at DESC) AS rn
  FROM orders
) t
WHERE rn = 1;

-- Month-over-month revenue change

       revenue - LAG(revenue) OVER (ORDER BY month) AS mom_change
FROM monthly;

-- Running total & 7-day moving average

       SUM(amount) OVER (ORDER BY day) AS running_total,
       AVG(amount) OVER (ORDER BY day
                         ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma7
FROM daily;
Aggregate vs window. GROUP BY returns one row per group; the same function OVER() returns one row per input row with the group stat attached. Use windows when you need detail and summary together.

CTEs & subqueries

A Common Table Expression (WITH … AS) names a query so you can build a pipeline of readable steps instead of nesting subqueries inward. CTEs make complex logic legible and let you reuse a result; recursive CTEs walk hierarchies and generate sequences. Subqueries still shine for quick scalar lookups and EXISTS checks.

sql
WITH paid AS (
    SELECT customer_id, amount FROM orders WHERE status = 'paid'
),
totals AS (
    SELECT customer_id, SUM(amount) AS spend FROM paid GROUP BY customer_id
)
SELECT c.name, t.spend
FROM   totals t
JOIN   customers c ON c.id = t.customer_id
WHERE  t.spend > 1000;

-- Correlated subquery: customers above their region's average
SELECT * FROM orders o
WHERE amount > (SELECT AVG(amount) FROM orders WHERE region = o.region);

-- EXISTS (often faster than IN for "has any match")
SELECT * FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id);

-- Recursive CTE: org hierarchy
WITH RECURSIVE tree AS (
    SELECT id, manager_id, 1 AS lvl FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.id, e.manager_id, t.lvl + 1
    FROM employees e JOIN tree t ON e.manager_id = t.id
)
SELECT * FROM tree;

Indexes & performance

An index is a sorted lookup structure (usually a B-tree) that lets the engine find rows without scanning the whole table — turning an O(n) scan into O(log n). Calculus & gradients

Training a model = minimizing a loss function. The derivative tells you the slope (how loss changes as a weight changes); the gradient generalizes this to many weights as the vector of partial derivatives, pointing in the direction of steepest increase. Gradient descent therefore steps in the opposite direction to go downhill. That single idea — follow the negative gradient — underlies essentially all model fitting.

Update rule, with learning rate \(\eta\):

text
θ_new = θ_old − η · ∇L(θ)

Written with the gradient symbol: \(\theta \leftarrow \theta - \eta\,\nabla_\theta L(\theta)\).

Sampling & the Central Limit Theorem

We rarely measure a whole population, so we estimate from a sample. The Central Limit Theorem is why statistics works: regardless of the population's shape, the distribution of the sample mean approaches a normal distribution as sample size grows, with standard deviation \(\sigma/\sqrt{n}\) (the standard error). This is what lets us build confidence intervals and run tests even on non-normal data.

Standard error. \(\mathrm{SE}=\dfrac{\sigma}{\sqrt{n}}\). Quadrupling the sample size only halves the error — diminishing returns. A 95% confidence interval is roughly \(\bar{x}\pm 1.96\,\mathrm{SE}\).

Inference & hypothesis testing

Hypothesis testing asks: could the effect I see be just noise? You assume a null hypothesis \(H_0\) (no effect), compute how surprising your data would be if \(H_0\) were true (the p-value), and reject \(H_0\) if that's below a threshold \(\alpha\) (commonly 0.05). The p-value is not the probability the hypothesis is true — it's \(P(\text{data this extreme}\mid H_0)\).

Multiple comparisons. Run 20 tests at \(\alpha=0.05\) and you expect one "significant" result by chance. Correct for it (Bonferroni, FDR) when testing many hypotheses.

Bayes & conditional probability

Bayes' theorem updates a belief in light of evidence — the backbone of Naive Bayes classifiers, spam filters, and the famous "how likely is disease given a positive test" puzzle. The counterintuitive lesson: with a rare condition, even an accurate test produces many false positives, so the posterior can be far lower than the test's accuracy.

\(P(A\mid B)=\dfrac{P(B\mid A)\,P(A)}{P(B)}\)

posterior = (likelihood × prior) / evidence

Base-rate example. A disease affects 1% of people; a test is 99% accurate. A positive result still gives only ~50% chance of disease, because the 1% sick are swamped by false positives from the 99% healthy. Always weight by the prior.

Combinatorics

Counting underlies probability and complexity. The core choice: does order matter? If yes, permutations; if no, combinations. If you can reuse items, multiply with repetition.

Permutations\(P(n,k)=\dfrac{n!}{(n-k)!}\)pick \(k\) ordered
Combinations\(\binom{n}{k}=\dfrac{n!}{k!(n-k)!}\)pick \(k\) unordered

Data leakage is the silent killer: any time information from the test set (or the future, or the target) sneaks into training, your metrics look great and fail in production. Fit scalers/encoders on the training fold only, then apply to validation/test.

Bias–variance & over/underfitting

Total error decomposes into bias (error from wrong assumptions — the model is too simple, underfitting), variance (error from sensitivity to the particular training data — the model is too complex, overfitting), and irreducible noise. You can usually trade one for the other; the art is finding the sweet spot where validation error is lowest.

\(\text{Error} = \text{Bias}^2 + \text{Variance} + \sigma^2_{\text{irreducible}}\)

SymptomDiagnosisFixes
High train error & high val errorUnderfitting (high bias)more features, complex model, train longer, less regularization
Low train error, high val errorOverfitting (high variance)more data, regularization, simpler model, dropout, early stopping
Both low & closeGood fit ✅ship it
Regularization penalizes complexity to curb variance. L2 (Ridge) shrinks weights smoothly toward zero; L1 (Lasso) drives some weights exactly to zero, doing feature selection; ElasticNet blends both. Larger penalty \(\lambda\) ⇒ simpler model ⇒ more bias, less variance.

Train/validation/test & cross-validation

You split data so you can estimate performance on unseen examples. Train fits parameters, validation tunes hyperparameters and picks models, and the test set is touched once at the very end for an honest estimate. k-fold cross-validation rotates the validation fold k times and averages — a more stable estimate that uses all data, essential when datasets are small.

Regression

Regression predicts a continuous value by fitting a line/surface that minimizes squared error. Linear regression assumes a linear relationship and is the interpretable baseline; its coefficients are the effect of each feature holding others fixed. Add polynomial terms for curvature, and regularize (Ridge/Lasso) when features are many or correlated.

\(\hat{y}=\beta_0+\beta_1x_1+\cdots+\beta_px_p,\qquad \text{minimize } \sum_i (y_i-\hat{y}_i)^2\)

MetricFormula ideaReading
MSE / RMSEmean (root) squared errorpenalizes big errors; RMSE in target units
MAEmean absolute errorrobust to outliers
R²variance explained1 = perfect, 0 = no better than mean
Linear regression assumptions: linearity, independence of errors, homoscedasticity (constant error variance), normally distributed residuals, little multicollinearity. Violations don't break prediction but invalidate the coefficient inference.

Classification

Classification predicts a category. Logistic regression — despite the name — is the linear classifier baseline: it passes a linear score through the sigmoid to output a probability, then thresholds it. Other staples: k-NN (vote among nearest neighbours, no training), SVM (maximum-margin boundary, kernels for nonlinearity), and Naive Bayes (fast, strong with text).

Sigmoid: \(\sigma(z)=\dfrac{1}{1+e^{-z}}\in(0,1)\), with \(z=\beta_0+\boldsymbol\beta^\top\mathbf{x}\)

Decision trees & ensembles

A decision tree splits the data on the feature that best separates the target (by Gini impurity or information gain), recursively — wonderfully interpretable but prone to overfitting. Ensembles combine many trees to fix that: bagging/Random Forest trains trees on bootstrapped samples in parallel and averages (reduces variance), while boosting (XGBoost, LightGBM, gradient boosting) trains trees sequentially, each correcting the last's errors (reduces bias). Boosted trees are the workhorse for tabular data.

Evaluation metrics

The single biggest classification mistake is trusting accuracy on imbalanced data: predict "not fraud" always and you're 99.9% accurate yet useless. The confusion matrix gives the honest building blocks; from it, precision asks "of those I flagged, how many were right?" and recall asks "of the real positives, how many did I catch?". Which you favour depends on the cost of each error type.

Neural networks & deep learning

A neural network stacks layers of weighted sums followed by nonlinear activation functions; depth lets it learn hierarchical features automatically, which is why deep nets dominate images, audio, and language. Training is gradient descent powered by backpropagation (the chain rule computing each weight's gradient). The practical knobs are architecture, activation, optimizer, learning rate, and regularization (dropout, weight decay, early stopping).

PieceRoleCommon choice
Activationadd nonlinearityReLU (hidden), softmax/sigmoid (output)
Losswhat to minimizecross-entropy (class), MSE (regression)
Optimizerhow to descendAdam (default), SGD+momentum
Regularizationfight overfittingdropout, weight decay, early stopping, batchnorm
Architecturematch the dataCNN (images), RNN/LSTM (sequences), Transformer (text)
Why nonlinearity matters. Stacking linear layers with no activation collapses to a single linear layer — the network could only fit lines. ReLU (\(\max(0,x)\)) and friends are what let depth model complex functions, and they keep gradients flowing (mitigating vanishing gradients). Transformers replaced recurrence with self-attention, letting models weigh every token against every other in parallel — the architecture behind modern LLMs.

CLIP, contrastive learning & multimodal retrieval

CLIP learns a shared embedding space for images and text by training two encoders so that matching image–text pairs land close together and mismatched pairs land far apart. That alignment is what enables zero-shot classification (compare an image to text prompts like "a photo of a cat") and multimodal retrieval (search images with text and vice versa) without task-specific training.

ConceptWhat it means
Two encodersImage encoder (ViT/CNN) + text encoder (Transformer) map each modality to a vector
Contrastive objectivePull matched pairs together, push the rest apart — the InfoNCE loss over a batch
Shared embedding spaceImage and text vectors are comparable directly via cosine similarity
Temperature \(\tau\)Scales the similarity logits; sharpens/softens the contrastive distribution
Zero-shot classificationEncode class names as prompts; pick the nearest to the image embedding
RetrievalEmbed the query, nearest-neighbour search (FAISS/ScaNN) over the index

Cosine similarity drives matching: \(\text{sim}(\mathbf{u},\mathbf{v})=\dfrac{\mathbf{u}\cdot\mathbf{v}}{\lVert\mathbf{u}\rVert\,\lVert\mathbf{v}\rVert}\)

labels = torch.arange(N)                           # the diagonal = correct pairs
loss = 0.5 * (cross_entropy(logits, labels)        # image -> text direction
            + cross_entropy(logits.T, labels))     # text -> image direction
# Zero-shot: similarity of one image vs. K text prompts -> argmax
Likely ML6 probes: "Why contrastive instead of a classifier?" (no fixed label set; learns transferable alignment). "How would you build retrieval over a million images?" (encode once, store vectors, approximate nearest-neighbour index, re-rank top-k). "Failure modes?" (domain shift from the training distribution, fine-grained distinctions, prompt sensitivity).

Deep learning & the attention mechanism

Aim for conceptual clarity, not derivations. Backpropagation is just the chain rule run backward through the network, reusing each layer's local gradient so every weight learns how it contributed to the loss. Self-attention lets each token look at every other token and weight them by relevance, computed in parallel — which is why Transformers scale better than RNNs that must process a sequence step by step.

Attention: \(\text{Attn}(Q,K,V)=\text{softmax}\!\left(\dfrac{QK^\top}{\sqrt{d_k}}\right)V\)

RNN/LSTMTransformer
Sequential — step t needs t−1Parallel — all positions at once
Long-range info fades (vanishing gradient)Direct token-to-token paths via attention
Hard to parallelize on GPUsHighly parallel ⇒ scales with data/compute
One-line answers to have ready. Q/K/V = "what I'm looking for / what I offer as a key / the content I pass on". The \(\sqrt{d_k}\) scaling keeps dot products from getting huge and saturating the softmax. Multi-head attention = several attention patterns learned in parallel, then concatenated. Positional encodings reinject order, since attention itself is permutation-invariant.

Working with APIs: pagination & rate limits

Extraction in real pipelines usually means pulling from REST APIs that return data in pages and cap how often you can call (rate limits). The robust pattern: loop until there's no next page, respect the limit by backing off, and retry transient failures with exponential backoff so a blip doesn't kill the whole run.

python
import requests, time

def fetch_all(url, params=None):
    params = dict(params or {}); results = []
    while url:
        r = requests.get(url, params=params, timeout=30)
        if r.status_code == 429:                       # rate limited
            wait = int(r.headers.get("Retry-After", 5))
            time.sleep(wait); continue                 # back off & retry
        r.raise_for_status()
        body = r.json()
        results.extend(body["data"])
        url = body.get("next")                         # cursor/next-link pagination
        params = None                                  # next URL already encodes them
    return results

# Exponential backoff for transient 5xx errors
def with_retry(fn, tries=5):
    for i in range(tries):
        try: return fn()
        except requests.RequestException:
            if i == tries - 1: raise
            time.sleep(2 ** i)                          # 1,2,4,8,16s
Pagination styles to recognize: offset/limit (?offset=100&limit=50 — simple but can skip/dupe if data changes), page number (?page=3), and cursor/keyset (an opaque "next" token — most reliable for large, changing datasets). Always make extraction resumable and idempotent.
